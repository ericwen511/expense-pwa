# 個人記帳PWA — 專案交接文件

給 Claude Code 接手開發用。以下都是已經和使用者討論並定案的決策，不需要重新確認。

## 專案目標

個人使用的記帳系統，網站+手機PWA，資料跨裝置同步。

## 技術棧（已定案）

- 前端：純網頁PWA（HTML/CSS/JS 或視需要用框架，第一階段用原生JS已驗證可行）
- 資料庫：Supabase（Postgres + Auth + RLS），理由是記帳資料本質關聯式（交易/分類/帳戶互相關聯），未來要做報表、多帳戶、家庭共享時比NoSQL好維護
- 部署：Vercel
- 版控：GitHub

## 工作流程偏好（重要，每次都要遵守）

**這個專案要連接 GitHub + Vercel，每次改完功能請自動 commit + push。**

## 已完成並驗證的部分

隨附的 `expense-pwa/` 資料夾是第一階段的技術驗證版本（不含Supabase），已確認：
- 可安裝到手機桌面（PWA manifest + service worker）
- 離線可開啟（靜態資源快取）
- IndexedDB本地儲存，刷新不丟失
- 響應式排版（手機/桌面皆可用，手機字體另外調校過大小）
- 核心功能：新增/刪除交易、分類管理（含商家管理）、搜尋、篩選、支出/收入切換

## 資料表設計（已定案，完整可執行SQL請直接用隨附的「supabase-建表SQL.sql」）

**重要：請直接執行 `supabase-建表SQL.sql` 這個檔案，不要重新設計schema，裡面已經整合了實測踩坑後才確定的規則，以下是背景說明：**

核心四表：`accounts`、`categories`（含parent_id子分類階層）、`merchants`（商家，PWA測試版才加的，之前的舊版設計文件沒有）、`transactions`。
兩個關鍵預留欄位：
- `transactions.client_generated_id`：離線同步去重用（unique constraint擋重複）
- `transactions.deleted_at`：軟刪除，避免多裝置同步時的tombstone問題

擴充表（第三、四階段，尚未寫入SQL檔，之後再補）：`budgets`、`tags`/`transaction_tags`、`attachments`。

**三個實測才學到、務必遵守的規則：**

1. **所有表都建在獨立的 `expense_app` schema底下，不要用預設的`public`**——因為同一個Supabase免費專案（Personal data）裡還放著另一個App（portable-ideas-app）的表，混在`public`底下會很亂。已在SQL檔第一行 `create schema if not exists expense_app;` 處理好。

2. **光有RLS政策不夠，一定要額外用GRANT明確授權角色**，不然會遇到 `permission denied` 或 `Invalid API key` 這類錯誤（這是2026年Supabase收緊的新規則，網路上大部分教學文章都還沒提到）。已在SQL檔最後面處理好，授權對象是`authenticated`角色（不是anon，因為使用者需要先登入）。

3. **有一步SQL做不到，需要人類手動去Supabase Dashboard操作**：Settings → API → 把 `expense_app` 加進「Exposed schemas」清單。這一步麻煩使用者本人手動完成，不是Claude Code能透過程式碼做到的部分，請在建表SQL執行完後提醒使用者去做這一步，並且在使用者確認做完之前，任何透過API對這些表的呼叫都會失敗（這不是程式碼寫錯，純粹是這個設定還沒生效）。

所有表都開了RLS，policy用 `auth.uid() = user_id`。

## 分階段規劃（已定案）

### 第一階段：基礎MVP（難度：低）— 目前隨附版本已完成技術驗證部分
- 記帳核心（快速記帳、支出收入切換、預設分類、列表分組、編輯刪除）
- 基礎統計（本月收支總覽、分類圓餅圖、日期區間篩選）
- PWA基礎能力（已驗證）
- 資料匯出入：**JSON做完整匯出+匯入（結構化備份還原）；CSV只做匯出，不做匯入**（CSV格式不標準，匯入去做欄位對應複雜度不划算）

### 第二階段：帳戶體系 + 雲端同步（難度：中，但同步機制偏中高）
- 帳戶管理：多帳戶、餘額（**動態計算**：initial_balance + 該帳戶所有交易加總，不用額外存快取餘額欄位）、轉帳（用transfer_to_account_id）、啟用/停用
- 雲端同步：Supabase Auth（Email+密碼，**bcrypt由Supabase內建處理，不用自己刻**）；**同步機制第一版用全量比對即可，不要一開始就做增量同步(updated_at)+正式衝突日誌，資料量小(個人一年頂多幾千筆)全量比對完全夠用，等真的覺得慢再優化**
- 資料安全：HTTPS、Postgres預設的靜態加密已足夠，不用額外加密層
- 進階篩選搜尋：多分類/帳戶/金額區間篩選、關鍵字搜尋、儲存常用篩選條件
- **建議順序**：先做完帳戶管理（不依賴雲端同步）並測試過，再接雲端同步，避免兩個一起改導致除錯困難
- 裝置管理降低優先度，先做「登出目前裝置」即可，不用做完整裝置清單管理

### 第三、四階段（尚未細部規劃，之後再談）
- 離線記帳自動同步、收據照片、預算提醒、多幣別、家庭共享等
