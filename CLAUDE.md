## 專案基本資訊

- 個人使用的記帳+財富管理系統，網站+手機PWA，資料跨裝置同步
- 前端：純原生 HTML/CSS/JS（無框架），單一 SPA，`index.html` + `app.js` + `styles.css`
- 資料庫：Supabase（Postgres + Auth + RLS），schema 全部建在 `expense_app` 底下（不用 `public`，避免跟同專案其他App混在一起）
- 部署：Vercel（GitHub push 自動部署）
- **工作流程慣例：每次改完功能，測試過後要自動 commit + push，不用額外確認**
- Supabase 的 SQL migration 檔案全部存放在 `docs/*.sql`，依執行順序命名，每個都是「一次性執行過就不用再跑」，已執行過的都會在檔案開頭註記狀態

## 四、跟原本規劃不同的技術決策

1. **記帳系統的外幣不自動換算成台幣總額**——總覽的支出/收入/結餘、分類圖表只計入台幣帳戶的交易，外幣交易不併入避免不同幣別金額直接相加造成誤導。這是跟使用者確認過的決策，不是遺漏。
2. **財富管家的現金/銀行類資產，不走 `assets` 表手動輸入，而是直接即時抓取記帳系統裡所有帳本的帳戶餘額**（跨帳本聚合，見 `app.js` 的 `sbGetAllTransactionsAllLedgers()` 跟 `computeWealthAccountBalance()`）。這跟最初 SQL 設計文件裡「assets.category 可以是 cash」的構想不同——現金餘額本來就已經在記帳系統即時準確地追蹤，沒必要在財富管家裡重複手動輸入一次，兩邊容易對不上。`assets` 表現在只用來放 investment/real_estate/precious_metal/crypto/insurance/other 這幾類需要「手動估值快照」的資產。
3. ~~財富管家跟記帳系統是兩套完全獨立的視覺主題（深色金色 vs 淺色綠色）~~ **【2026-07-12更新，此決策已推翻】**：原本刻意做成兩套獨立主題，但實際使用後使用者反應兩邊切換時明暗反差太強烈（淺色↔深色）。改成記帳系統也套用跟財富管家一致的深色金色主題（`styles.css` 的 `:root` 變數：`--bg`/`--card`/`--text-*`/`--border` 改深色系，`--primary`/`--primary-dark`/`--primary-light` 從綠色改成跟財富管家一樣的金色 `#C9A25D`）。原因：深綠色 `--primary` 在深色背景上對比不足、文字難辨識，且使用者認為金色在深色背景上效果本來就比較好，兩邊統一也比較舒服。兩邊各自導覽列有一個入口可以切換到對方，這點不變。
4. **定期定額交易採用「預先產生未來12個月的真實交易列」**，不是畫面上疊加一層預覽、不寫進資料庫。原因：使用者要能直接在交易列表看到未來排定的收支、也要能直接編輯/刪除單筆，用真實資料列最直接。用日期比對（未來日期淺灰顯示、日期一到自動變正常色）取代额外的狀態欄位，不需要背景排程。
5. **每月預算先只做單一總預算，不分類別**——是跟使用者確認過的範圍決策，之後如果要分類別預算，`budgets` 表可以再擴充。
6. **股票即時股價需要新增 Supabase Edge Function 當代理**，這是超出原本「純前端PWA＋Supabase資料庫」架構的新基礎設施。原因是**實測**（不是理論猜測）瀏覽器直接呼叫台股 TWSE 或美股資料源，都會被 CORS 擋掉（`Failed to fetch`），只能透過伺服器端（Edge Function 對伺服器呼叫沒有 CORS 限制）代理才能真正抓到股價。陸股目前沒有找到穩定資料源，先讓使用者手動輸入。
   - **美股資料源後來從 Stooq 改成 Yahoo Finance（2026-07-12）**：原本選用的 Stooq 免費CSV端點，實測時發現已失效——`/q/l/` 即時報價端點回傳純 404（伺服器端已下架/搬移，不是CORS或程式問題），備用的 `/q/d/l/` 下載端點則被新增的 anti-bot JS挑戰頁保護，Deno Edge Function 的純伺服器端 fetch 無法通過（沒有瀏覽器JS執行環境可以解題）。改用 Yahoo Finance 的 `query1.finance.yahoo.com/v8/finance/chart/{symbol}` 端點取代，實測穩定可用。
7. **JSON匯入的錯誤處理原則從「一筆錯誤就整批中斷」改成「逐筆記錄失敗、繼續處理其餘資料」**——這是在匯入使用者真實歷史資料時發現的嚴重bug（原本的寫法會導致匯入 2321 筆資料時，第一筆網路瞬斷就整批放棄，只成功不到5%），修正後所有匯入/批次寫入迴圈都遵循「單筆失敗不影響整批」的原則。
